## 4.2 프로젝트 회고

### 잘 된 점
- View·Service·API·DB·common 계층 분리가 명확해 기능 추가·유지보수가 수월했다.
- STT 응답이 깨지거나 잘려 와도 정규식으로 세그먼트를 복구하는 fallback 파싱(`SttResponse.fromJson`)을 구현해 안정성을 높였다.
- API 재시도·타임아웃, 트랜잭션 기반 저장, 사용자별 기록 분리 등 견고함을 갖췄다.

### 아쉬운 점 / 개선 사항

- **① 실시간 변환(라이브 전사)을 구현하지 못한 점**
  현재 구조는 *녹음 종료 → `변환 시작` 클릭 → 녹음 파일 전체를 STT API로 전송 → 결과 수신* 으로 이어지는 **일괄(batch) 처리** 방식이다. 그래서 말하는 즉시 텍스트가 채워지지 않고, **녹음이 끝나야 비로소 변환 결과가 화면에 반영**된다는 한계가 있다.
  → **개선 방향**: 오디오를 짧은 구간(chunk)으로 잘라 전송하는 **스트리밍 STT(WebSocket 기반 partial result)** 방식으로 전환하면, 회의·온라인 강의 도중에도 발화와 거의 동시에 자막이 갱신되는 실시간 전사가 가능하다.

- **② 데이터베이스를 별도 서버로 분리하지 못한 점**
  변환 기록이 프로그램을 실행하는 **바로 그 PC의 로컬 SQLite 파일(`speechnote.db`)** 에만 저장된다. 계정별로 기록을 나누긴 했지만 결국 **그 한 대의 컴퓨터 안에서만 유효**하므로, "다른 기기에서 같은 계정으로 이어 보기"나 "기기 고장·교체 시 데이터 보존" 같은 다중 사용자 서비스의 이점이 사실상 발휘되지 못한다.
  → **개선 방향**: 기록을 **중앙 DB 서버(클라우드 MySQL/PostgreSQL)나 백엔드 API** 에 저장하도록 바꾸면, 어느 기기에서 로그인하든 같은 기록을 보고 백업·공유까지 할 수 있다. *(이번 프로젝트는 서버 인프라·비용 제약으로 로컬 SQLite로 범위를 한정했다.)*
